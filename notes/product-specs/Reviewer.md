# Reviewer (an Analysis Subsystem)
Reviewer is what I'm calling the subsystem that reviews agreements and creates findings.

# Overall Structure
A user creates a Matter, which is an overall umbrella concept for one agreement.

Within a Matter, a user will upload a contract, supplemental documents, and answers to questions, and generate metadata.

## Review
Within a Matter, a user may request a Review. A Review is a collection of Findings that are generated by "Lens" prompts that are run against the Matter. There will probably be multiple types of reviews, eg Quick Review, Full Review, Risk Review, etc.

A Review has two categories of responsibility:
1. Support the Web UX
This is the main structure that feeds the web views, reports, and user interaction.

2. Be the doorway to the Review process
Be the main point of contact between the user/FE and the async processes that consist of running the review.


## ReviewRun
A ReviewRun -- aka "run" -- is one multi-part process that runs on the queue. Different types of Reviews will have unique run patterns, but I think this will be typical:

1. Run one prompt that sets up the review system prompt, the contract and supplemental documents, and metadata. This gets stored as a prefix using Anthropic's cache mechanism.

2. Run a set of lenses (maybe 6 or 12). Each lens uses the prefix and is a prompt that examines the documents in a different way. Each lens will return findings.

3. As the Lens is returning Findings, the system will return small updates to show the user. Like "4 findings", "5 findings"...

4. The FE will be polling the Review to find the status of the run. This will be a granular status for all the lenses and the overall run.

5. The findings get stored against the Review as each lens finishes.


## Failure and Retry
1. If the initial prompt fails, it must be retried.
2. If an individual lens fails, it must be retried.
3. We should have 2 LLM vendors configured. Vendor A is Azure, Vendor B is Anthropic.
4. If, after 2 tries, the initial prompt fails, we switch the whole run to B.
5. If an individual lens fails after 2 tries on A, we switch that lense to B.
6. There are specific errors that indicate that the vendor is having trouble. For these errors, we switch to the next vendor.
7. If the errors are not related to vendor, we try N times and then fail.


## Technical Stuff
1. Each lens will return JSONL.
2. If a lens fails, it will be retried.
3. There may be different shapes of findings, depending on the lens.
4. This is an embryonic product, so we are optimizing for flexibility. JSONB is our friend.
5. Prompts will be stored as go templates in the git repo. They will be uploaded to the database for runtime.
6. River will be the queue.
